# opencode-fusion

[English](README.md)

一个 [OpenCode](https://opencode.ai) 的多模型融合插件：让前沿模型负责关键决策，让廉价模型承担机械执行。

灵感来自 Cognition 的 [Devin Fusion](https://cognition.com/blog/devin-fusion)——sidekick 架构来自他们的工作。Goal 模式和 reviewer 循环演化自 [my-codex](https://github.com/faceair/my-codex)，一套早期的 Codex 工作流配置。这个插件把这些思路整合到 OpenCode，作为开源插件用你自己的模型在本地运行。

## 为什么需要它

用单模型 agent 做真实工程任务，你大概率遇到过这些问题：

- **贵。** 前沿模型烧 token 跑 `grep`、`cat`、`npm test`，就是在烧钱。但全路由到便宜模型，你的架构决策就交给了一个不理解 trade-off 的模型。
- **Cache miss。** "智能助手"类工具让一个模型去问另一个模型——但每次跨模型调用都丢失上下文缓存，你得重新付一遍完整 prompt 的费用。长任务里这笔账涨得很快。
- **盲区。** 一个模型写代码、审代码、批准代码。没有独立视角来发现你没考虑到的边界情况。
- **停滞。** agent 做完一步就停。你打"继续"。它再做一步又停。两小时的任务要六小时盯着。
- **失忆。** 上下文压缩把计划抹掉了。agent 忘了自己在做什么，从头开始。

opencode-fusion 就是围绕这些问题设计的。

## 工作原理

三个 agent，各自有独立的模型、上下文和职责：

```
┌─────────────────────────────────────────────────┐
│  fusion（前沿模型）                              │
│  负责：计划、歧义解释、最终审查、goal            │
│                                                 │
│  通过 task() 委托 ─────────────────┐            │
│  通过 task() 咨询 ─────────────┐   │            │
│                               ▼   ▼            │
│                   ┌──────────────┐ ┌─────────┐  │
│                   │ reviewer     │ │ sidekick│  │
│                   │（只读）       │ │（廉价） │  │
│                   │ 架构审查      │ │ 执行    │  │
│                   │ + 对抗式审查  │ │ 发现    │  │
│                   │              │ │ 验证    │  │
│                   └──────────────┘ └─────────┘  │
└─────────────────────────────────────────────────┘
```

### 1. Sidekick 委托

前沿模型只管判断，廉价模型承担机械工作。

这不是一个给整个任务选一个模型的路由器。fusion **动态**委托——它逐步决定哪个 agent 做什么：

- **交出慢验证。** sidekick 跑 90 秒的 Playwright 测试套件，fusion 同时去做下一个决策。测试密集型任务成本降 60%+，质量零损失。
- **收回判断密集的工作。** sidekick 撞到决策点——API 形状、错误语义、跨模块边界——fusion 把那个里程碑收回来，不让便宜模型猜。
- **定向跟进。** sidekick 发现了意外情况？fusion 发一个聚焦的问题回去，而不是自己重新读代码。

两个 agent 各自保持持久缓存上下文。委托不触发 cache miss——这是跟"问另一个模型"工具的关键区别。

### 2. 独立审查

第三个 agent 作为只读子 agent 运行，从不编辑文件。fusion 在两种场景下咨询它：

**实现前**——当变更高风险时：
- 共享 API 契约、公共接口、跨子系统边界
- 生命周期、并发、持久化、schema 语义
- 安全、凭据、隐私、生产关键路径
- 同一方案反复失败
- 本地验证后信心仍然不足

**交付前**——对任何非平凡变更，reviewer 做 code review：正确性、完整性、回归、架构一致性。对高风险变更，它还执行**对抗式审查**——从攻击者视角探测 diff：

- *如果这个输入是 50MB 而不是 5KB 会怎样？*
- *如果时间戳来自未来会怎样？*
- *如果后台 worker 执行到一半被杀掉然后重试会怎样？*
- *如果两个用户同时提交相同请求会怎样？*

每个发现都追踪完整路径：入口 → 处理 → 存储 → 输出 → 副作用。这种审查能在生产事故发生之前发现它们，而不是之后。

### 3. Goal 模式 + 自动续命

每个委托任务都会创建一个 goal：一句目标，一段简短计划。goal 持久化到磁盘，能扛住两个通常会打断任务推进的情况：

**上下文压缩。** 当 OpenCode 压缩会话（长任务会自动触发），恢复上下文里只保留 subagent `task_id`。goal 本身通过 tool result 历史和 `get_goal` 工具恢复，让 goal 对模型的可见性和 todo 列表保持一致。

**进程重启。** goal 存在本地 JSON 文件里。杀进程、重启机器、明天再来——`opencode run --continue` 拿到同一个 session ID，goal 还在那里。

goal **自动续命**直到显式关闭。agent 不会做完一步就停下来等你打"继续"。它持续推进：用内置 todo 工具更新里程碑，把下一块工作委托给 sidekick，审查结果，只在工作验证完成或遇到明确 blocker 时才关闭 goal。

### 4. 第一性原理 + 对抗式审查

两种思维模式内嵌在 agent 提示词里——不是可选开关，而是默认行为：

**第一性原理。** 遇到 bug 修复、架构决策或方案选择时，agent 从基本事实和约束推导，而不是去训练数据里找最像的模式。区别在于：

- *症状修复：* "OpenAI 的抓取坏了，修一下 fetcher。"（下周同样的 bug 又回来了）
- *根因修复：* "流量路由层有一个潜在的失败模式，fetcher 只是第一个受害者。修路由。"（这类 bug 被彻底消除）

**对抗式审查。** 对涉及不可信输入、持久化、外部内容或后台 worker 的变更，reviewer 走过每个输入路径并问：如果这个输入是极端的、畸形的或恶意的会怎样？超大 payload、未来时间数据、无限循环、竞态条件、注入路径——从入口追踪到影响，每个发现都给出具体修复建议。

## 适用场景

- **长任务**——需要跨越上下文压缩和进程重启而不丢失线索
- **复杂 debug**——明显修复治标不治本
- **判断密集的多阶段重构**——关键决策不该交给便宜模型
- **高风险变更**——交付前需要独立审查
- **开放性工作**——性能优化、根因排查、架构清理——最佳下一步无法预先规划，需要 reviewer 循环帮助收敛

如果你只想要一个轻量助手随聊随停，这套配置可能不是最合适的。

## 安装

将 npm 插件添加到 OpenCode 配置（`~/.config/opencode/opencode.jsonc`）：

```jsonc
{
  "plugin": [
    [
      "@faceair/opencode-fusion",
      {
        "sidekick": {
          "model": "provider/model-name",
          "variant": "medium"
        },
        "reviewer": {
          "model": "provider/model-name"
        }
      }
    ]
  ],
  "default_agent": "fusion"
}
```

保存后重启 OpenCode。OpenCode 会在启动时自动安装 npm 插件。

| 选项 | 适用 agent | 说明 |
|------|-----------|------|
| `model` | sidekick, reviewer | 模型，格式为 `provider/model-id` |
| `variant` | sidekick, reviewer | 推理强度（`low`、`medium`、`high`、`xhigh`） |
| `options` | sidekick, reviewer | provider 特定选项 |

如果省略 `model`，agent 继承当前会话的模型。如果不想把 Fusion 设为默认 agent，可以省略 `default_agent`，需要时手动选择 `fusion` agent。
